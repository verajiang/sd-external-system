# 智播星平台管理后台 PRD

版本：V0.1  
日期：2026-06-16  
状态：基于当前管理后台原型与商家端 PRD 整理，供产品、设计、前端、后端评审

## 1. 执行摘要

智播星平台管理后台面向智播星内部运营、客服和实施人员，用于管理商家组织、开通商家服务、创建商家的第一个管理员账号，并维护商家的套餐、席位、额度、到期时间和登录状态。

后台管理的是“商家组织层”。商家内部的普通账号、成员额度、素材、生成记录和团队协作，仍由商家端管理员在商家端完成。

## 2. 证据来源

本 PRD 基于以下已实现原型和需求材料整理：

- 商家端 PRD：`docs/智播星商家端PRD.md`
- 后台入口与商家端入口隔离：`src/App.tsx`
- 管理后台原型页面：`src/pages/PlatformAdminPage.tsx`
- 平台后台数据模型：`src/types.ts`
- 平台后台 Mock 数据：`src/data/mock.ts`
- 单 HTML 交付脚本：`scripts/make-standalone.mjs`

已确认的业务依据：

- 商家端已有管理员 / 普通账号角色。
- 商家端团队页负责添加商家内部成员、生成初始密码、重置密码、管理席位和成员额度。
- 平台后台需要解决的是新增商家和创建商家第一个管理员账号，不应替代商家端团队管理。
- 商家状态只保留“启用 / 已停用”；“是否首次登录”由首管理员账号状态表达。
- 首管理员账号必须使用手机号作为登录账号。
- 初始密码只展示一次；关闭后如需再次获得，必须重置密码。
- 停用商家会立即中断该商家全部成员登录。
- 额度需要区分套餐额度、赠送额度和手工调整额度。
- 即将到期阈值固定为 15 天。
- 操作日志需要支持导出。

## 3. 用户与目标

### 3.1 平台运营 / 客服实施

核心目标：

- 查看所有商家组织。
- 新增商家并完成首管理员账号创建。
- 复制初始密码并交付给商家联系人。
- 调整商家的套餐、席位、额度和到期时间。
- 停用或启用商家。
- 重置商家首管理员账号密码。
- 查看关键操作日志，便于追踪后台变更。

### 3.2 商家联系人

商家联系人不是后台用户，但会接收平台运营交付的首管理员账号和初始密码。

商家联系人拿到账号后，进入商家端登录，并在商家端继续添加团队成员。

## 4. 信息架构

一级模块：

- 商家管理
- 操作日志

不做：

- 商机 / CRM 管理
- 合同、支付、发票
- 真实鉴权
- 商家内部成员的完整管理
- 素材、真人形象、生成任务的直接编辑
- 独立数据看板

## 5. 核心流程

### 5.1 新增商家与首账号创建

1. 平台运营进入“商家管理”。
2. 点击“新增商家”。
3. 填写商家名称、联系人、联系手机号。
4. 设置套餐、席位数、首笔套餐额度和首笔套餐额度有效期。
5. 填写首个管理员账号姓名和登录手机号。
6. 提交后系统创建商家组织。
7. 商家状态为“启用”。
8. 首管理员账号状态为“待首次登录”。
9. 系统生成初始密码。
10. 平台运营复制初始密码并交付给商家联系人。
11. 操作日志记录“开通商家”。

验收标准：

- 新增商家后，商家列表立即出现该商家。
- 新增商家后，商家详情可查看首管理员账号。
- 初始密码弹窗必须展示账号和密码，并提供复制按钮。
- 初始密码只展示一次；关闭弹窗后不再支持查看原密码，只能重置。
- 首管理员账号必须使用手机号，不支持邮箱作为登录账号。
- 商家状态不出现“待开通”。
- 首账号未登录状态使用“待首次登录”表达。

### 5.2 商家列表与筛选

商家管理页展示：

- 商家总数
- 启用商家数
- 15 天内到期商家数
- 额度不足商家数
- 商家名称
- 联系人和手机号
- 商家状态
- 套餐
- 席位使用
- 当前可用额度
- 到期时间
- 首账号状态

支持：

- 按商家名称、联系人、手机号、首账号搜索。
- 按“全部 / 启用 / 已停用”筛选。
- 导出当前筛选结果。
- 点击商家进入右侧详情。

验收标准：

- 商家状态只允许“启用 / 已停用”。
- 额度不足商家需要在列表中有视觉提示。
- 即将到期统计阈值固定为 15 天。
- 导出商家列表时，应导出当前筛选结果，而不是强制导出全量数据。
- 已停用商家仍可在列表中查看，但应清楚标识。

### 5.3 商家详情

商家详情展示：

- 商家名称
- 创建时间
- 当前状态
- 联系人
- 套餐
- 当前可用额度
- 额度来源汇总：套餐额度、赠送额度、手工调整额度
- 额度批次列表：来源、发放额度、剩余额度、有效期、备注
- 到期时间
- 已用席位 / 总席位
- 本月生成次数
- 本月消耗额度
- 生成成功率
- 首管理员账号姓名、账号、状态、最近登录时间

支持操作：

- 调整套餐 / 席位
- 新增额度批次
- 重置首账号密码
- 停用商家
- 启用商家

验收标准：

- 停用商家后，商家状态变为“已停用”，首账号状态变为“停用”，并立即中断该商家全部成员登录。
- 启用商家后，商家状态变为“启用”，首账号状态变为“启用”。
- 重置密码后展示新的临时密码，并写入操作日志。
- 调整套餐 / 席位后，详情和列表的套餐、席位同步更新。
- 新增额度批次后，详情和列表的当前可用额度同步更新。

### 5.4 套餐与席位变更

平台运营可调整：

- 套餐：试用版、基础版、专业版、旗舰版。
- 总席位数。

套餐升级 / 降级规则：

- 升级套餐只更新商家当前套餐配置，不自动改写历史额度批次。
- 如果升级套餐需要补发套餐额度，应另行新增一笔“套餐额度”批次，并设置该批次有效期。
- 降级套餐只更新商家当前套餐配置，不自动回收历史未过期额度。
- 如果降级需要扣减额度，本期原型不做负向额度批次；真实系统应通过后端额度账务规则处理。

席位增购 / 减购规则：

- 增购席位只增加总席位数，不新增额度批次。
- 减购席位只减少总席位数，但不得低于当前已用席位数。
- 已用席位不因调整总席位而丢失。

验收标准：

- 调整套餐 / 席位后，商家列表和详情同步更新。
- 操作日志记录调整后的套餐和总席位数。
- 套餐 / 席位变更不直接新增、覆盖或删除额度批次。

### 5.5 额度批次新增

平台运营可新增一笔额度批次。

每笔额度批次包含：

- 来源：套餐额度 / 赠送额度 / 手工调整额度。
- 发放额度。
- 剩余额度。
- 有效期。
- 备注。

验收标准：

- 新增后商家列表和详情同步更新。
- 当前可用额度 = 所有未过期额度批次的剩余额度之和。
- 不同额度批次允许拥有不同有效期。
- 新增额度不覆盖历史额度批次。
- 操作日志记录新增额度来源、新增额度数量和本批次有效期。

### 5.6 操作日志

操作日志展示：

- 时间
- 操作人
- 商家
- 动作
- 详情

支持：

- 导出日志

当前动作类型：

- 开通商家
- 停用商家
- 启用商家
- 调整套餐席位
- 新增额度批次
- 重置首账号密码

验收标准：

- 每次新增商家、停启用、调整套餐席位、新增额度批次、重置密码，都新增一条日志。
- 日志按时间倒序展示。
- 日志支持导出。
- 日志只做查看和导出，不提供编辑和删除。

## 6. 数据模型需求

### 6.1 PlatformOperator

平台后台用户：

- name：操作人名称。
- role：平台运营 / 客服实施。

### 6.2 ManagedMerchant

商家组织：

- id
- companyName
- contactName
- contactPhone
- status：启用 / 已停用
- plan
- seatsUsed
- seatsTotal
- creditGrants
- remainingCredits
- usedCredits
- expiresAt
- createdAt
- primaryAccount
- monthlyGenerations
- successRate

### 6.3 CreditGrant

额度批次：

- id
- source：套餐额度 / 赠送额度 / 手工调整额度
- amount：本批次发放额度
- remaining：本批次剩余额度
- expiresAt：本批次有效期
- createdAt：发放时间
- note：备注

计算规则：

- 当前可用额度 = 所有未过期额度批次的 remaining 之和。
- 每次增加额度都新增 CreditGrant，不直接覆盖商家总剩余额度。
- 额度消耗应优先扣减更早到期的未过期批次，避免临期额度沉淀。

### 6.4 MerchantPrimaryAccount

商家首管理员账号：

- name
- contact
- status：待首次登录 / 启用 / 停用
- lastLoginAt

约束：

- contact 必须是手机号。
- 初始密码只展示一次，不作为可查询字段长期展示。

### 6.5 PlatformAuditLog

后台操作日志：

- id
- time
- operator
- merchantName
- action
- detail

## 7. 权限与边界

平台后台：

- 可以新增商家。
- 可以创建商家首管理员账号。
- 可以调整商家套餐、席位，并新增额度批次。
- 可以停用 / 启用商家。
- 可以重置商家首管理员密码。
- 可以查看操作日志。
- 可以导出操作日志。

商家端：

- 商家管理员继续管理企业内部成员。
- 商家管理员继续管理素材、真人形象、授权记录、生成记录和成员额度。
- 普通账号继续只使用自己有权限的生成和资产能力。

后台不直接承担：

- 商家内部普通账号创建。
- 商家内部成员额度细分。
- 商家素材审核和真人授权处理。
- 真实支付或合同状态流转。

## 8. 非功能需求

- 技术栈：React + TypeScript + Vite + Tailwind CSS。
- 第一版不接真实后端，全部使用 mock data。
- 后台入口：`?app=admin` 或后台单 HTML。
- 默认入口仍为商家端，不影响商家端登录和页面。
- 构建后输出可本地打开的单 HTML。
- 视觉规范沿用商家端：白底运营界面、1px 浅灰边框、4px 圆角、主色 `#2461fb`、PingFang SC 14/22。
- 当前目标桌面端，不要求移动端适配。

## 9. 成功指标

产品验证指标：

- 商家开通完成率：新增商家后成功交付首账号的比例。
- 首账号首次登录率：首管理员账号从“待首次登录”变为“启用”的比例。
- 开通耗时：从录入商家到复制初始密码的平均耗时。
- 运营变更可追踪率：套餐、席位、额度批次、停启用、重置密码是否都有日志。
- 到期和低额度处理率：15 天内到期、额度不足商家被运营处理的比例。

原型验收指标：

- 默认商家端入口不受影响。
- `?app=admin` 可进入平台管理后台。
- 后台单 HTML 可本地打开。
- 新增商家、复制初始密码、导出商家列表、新增额度批次、停启用、重置密码、操作日志导出均可演示。

## 10. 风险与依赖

### 10.1 权限风险

平台后台具备商家组织级管理能力，真实落地时必须有后台用户鉴权和操作权限。

缓解：

- 后端按平台角色校验接口权限。
- 高风险操作写入不可篡改日志。
- 停用、重置密码等操作增加二次确认。

### 10.2 商家端边界风险

如果后台继续扩展商家内部成员管理，可能与商家端团队页能力重叠。

缓解：

- 平台后台只创建首管理员账号。
- 普通账号新增、成员额度、成员停启用继续放在商家端团队页。

### 10.3 套餐额度一致性风险

后台调整套餐 / 席位或新增额度批次后，商家端必须同步展示并参与生成额度判断。

缓解：

- 后端以商家组织额度批次数据为唯一来源，并保留每笔额度的来源、剩余额度和有效期。
- 商家端只消费后端返回的额度和席位数据。

## 11. 已确认规则

1. 首管理员账号必须使用手机号。
2. 初始密码只展示一次，关闭后必须通过重置密码再次生成。
3. 停用商家会立即中断该商家全部成员登录。
4. 额度必须按批次管理，并区分套餐额度、赠送额度、手工调整额度；每笔额度可以拥有不同有效期。
5. 即将到期阈值固定为 15 天。
6. 操作日志支持导出。

### M1：静态原型确认

- 确认商家状态只保留“启用 / 已停用”。
- 确认新增商家和首账号交付闭环。
- 确认后台与商家端团队页边界。
